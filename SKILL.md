# HTML 汇报 Skill

零依赖、动画丰富的 HTML 演示文稿生成工具，专为中文工作汇报场景设计。一次生成，浏览器打开即可全屏演示。

本 skill 基于 `zarazhangrui/frontend-slides` 改造，保留上游 MIT License 版权声明（Copyright (c) 2025 Zara Zhang），并面向中文工作汇报场景新增排版、交互、内容可靠性和质量验证规则。

## 关键词索引（复现校验）
- Phase 0 — 模式检测
- 模式 C — MD 文档转换
- 密度 — 内容密度驱动（v1.6）
- 禁止渲染 — 预览规则（v1.6）
- center-v — 垂直居中补偿
- section-kicker — section-head 模式
- validate.py — 质量验证

## 核心原则
1. **零依赖** — 单 HTML 文件，所有 CSS/JS 内联。无需 npm，无需构建工具。
2. **所见即所选** — 生成可视化预览，而非抽象的文字选项。用户靠"看"来做选择，而非"想"。
3. **有识别度的设计** — 杜绝"AI 泛审美"。每个演示文稿都像定制作品。
4. **视口适配（不可妥协）** — 每张幻灯片必须精确适配 100vh。幻灯片内绝不可出现滚动条。内容溢出？拆成多张。
5. **内容密度自适应排版** — 不同的内容类型（表格、图文、稀疏内容、卡片）采用不同的排版策略，不留大片空白。详见 [排版规范.md](references/排版规范.md)。<br>**垂直对齐补偿**：当一张幻灯片内容较少（只占视口上半部分），`.slide-content` 使用 `justify-content: center` 垂直居中，避免内容贴顶、底部大片空白。

## 全局阶段进度协议

所有模式都必须让用户第一时间知道当前处于哪个 Phase。提示只出现在对话中，禁止写入预览或最终 HTML。

- 新建/Markdown：`Phase 0 → 1 → 2 → 3 → 5 → 6（可选）`
- PPT：`Phase 0 → 4 → 2 → 3 → 5 → 6（可选）`
- 已有 HTML 增强：`Phase 0 → 3 → 5 → 6（可选）`

**阶段进入提示**必须在执行阶段首个动作前发送：

```text
[阶段进度] Phase X · 阶段名称
当前目标：<本阶段要完成什么>
完成标志：<满足什么条件才离开>
下一确认点：<用户下一次需要确认什么；没有则写“无”>
```

**阶段完成提示**必须在进入下一阶段前发送：

```text
[阶段完成] Phase X · <已完成结果>
阻塞：<无 / 明确问题>
下一阶段：Phase Y · 阶段名称
```

返工或重选时必须写明 `[阶段回退] 返回 Phase X · <原因>`。不要在同一 Phase 的每个内部动作后重复状态。

## 设计审美

避免泛化、"分布式"的 AI 输出。在网页设计中，这类输出被用户称为"AI 泛审美"。请做出有创意、独特的设计，让人眼前一亮。

关注重点：
- **字体排版：** 选择美观、独特、有趣的字体。中文字体必须加入回退链（Noto Sans SC / Source Han Sans SC / Microsoft YaHei）。禁止只用系统默认字体。详见 [风格预设.md](references/风格预设.md) 中的中文字体回退策略。
- **配色与主题：** 坚持统一的美学风格。使用 CSS 变量保持一致性。主色鲜明 + 强调色锐利，胜过均匀分散的配色方案。
- **动效：** 使用动画提升效果和微交互。优先 CSS 动画方案。聚焦高冲击力时刻：一个精心编排的页面入场 + 错落入场（animation-delay）能创造更多惊喜。
- **背景：** 营造氛围和深度，而非默认纯色。叠加 CSS 渐变、使用几何图案、或添加与整体风格匹配的语境效果。  避免 AI 泛审美：
- 过度使用的字体（Inter、Roboto、Arial、系统字体）
- 陈词滥调的配色（尤其是白色背景上的紫色渐变）
- 可预测的布局和组件模式
- 缺乏语境特征的千篇一律设计  发挥创意，做出非预期的选择，让作品感觉是为此场景精心定制的。在深色/浅色主题、不同字体、不同美学之间切换变化。

## 视口适配规则

以下规则适用于每个演示文稿的每一张幻灯片：
- 每个 `.slide` 必须设置 `height: 100vh; height: 100dvh; overflow: hidden;`
- **所有**字体大小和间距必须使用 `clamp(最小值, 首选值, 最大值)` — 禁止使用固定 px/rem
- 内容容器需要 `max-height` 限制
- 图片：`max-height: min(50vh, 400px)`（架构图和流程图可适当增加到 55-60vh）
- 必须为以下高度提供断点：700px、600px、500px
- 必须包含 `prefers-reduced-motion` 支持
- CSS 函数前不能直接加 `-` 号 — 使用 `calc(-1 * clamp(...))` 代替

**生成时，务必读取 [视口适配.css](references/视口适配.css) 并将其完整内容包含在每个演示文件中。**

### 每张幻灯片的内容密度限制
| 幻灯片类型 | 最大内容量 |
|-----------|-----------|
| 标题页 | 1 个标题 + 1 个副标题 + 可选标签行 |
| 内容页 | 1 个标题 + 4-6 条要点 **或** 1 个标题 + 2 个段落 |
| 功能卡片网格 | 1 个标题 + 最多 6 张卡片（2x3 或 3x2） |
| 代码页 | 1 个标题 + 8-10 行代码 |
| 引用页 | 1 句引用（最多 3 行）+ 署名 |
| 图片页 | 1 个标题 + 1 张图片（按排版策略选择约束） |
| 表格页 | 1 个标题 + 1 张表格（列数≥5或行数≥10时调整字号） |
| 图文混排页 | 1 个标题 + 左文字右图片的双栏布局 |

**内容超出限制？拆分成多张幻灯片。绝不塞满，绝不出现滚动条。**

**内容不足以填满？** 当一张幻灯片只有 section-head + ≤4 张卡片、或一段话 + 短列表时，`.slide-content` 加上 `center-v` class，整体垂直居中，避免内容贴顶、底部大片留白。

## Phase 0：模式检测

判断用户的需求：
- **模式 A：新建汇报** — 从零开始创建。进入 Phase 1。
- **模式 B：PPT 转换** — 将 .pptx 文件转为 HTML。进入 Phase 4。
- **模式 C：MD 文档转换** — 将 Markdown 文档转为 HTML 汇报。这是最常见的模式。自动识别 MD 中的表格、图片、标题层级，进入 Phase 1-内容识别（增强版）。
- **模式 D：已有 HTML 增强** — 改进现有的 HTML 演示文稿。读取它、理解它、增强它。**遵循模式 D 的修改规则。**  **依赖检查（进入模式前必须执行）：**
```bash
# PPT 转换模式（模式 B）需要 python-pptx python -c "from pptx import Presentation" 2>/dev/null && echo "✓ python-pptx 已安装" || echo "✗ python-pptx 未安装，请执行: pip install python-pptx"

# PDF 导出（Phase 6B）需要 Node.js + Playwright command -v npx >/dev/null 2>&1 && echo "✓ Node.js 已安装" || echo "✗ Node.js 未安装，请访问 https://nodejs.org 安装"
```
如果依赖缺失，**立即告知用户**安装命令并等待用户确认后再继续。不要假设用户已经安装了依赖。
**智能感知：** 在模式检测中同时感知用户意图：
- 用户提到"弹窗"、"点击查看详情"、"展开" → 默认启用"标准"交互
- 用户提到"长文档"、"在线浏览"、"不要翻页" → 默认启用"连续滚动"模式

### 模式 D：修改规则

增强已有演示文稿时，视口适配是最大风险：
1. **添加内容前：** 统计现有元素数量，对照密度限制检查
2. **添加图片：** 必须有 `max-height: min(50vh, 400px)`。如果幻灯片内容已达上限，拆成两张
3. **添加文字：** 每张最多 4-6 条要点。超出限制？拆成延续页
4. **每次修改后验证：** `.slide` 有 `overflow: hidden`，新元素使用 `clamp()`，图片有视口相对 max-height，内容在 1280x720 分辨率下不溢出
5. **主动重组：** 如果修改会导致溢出，自动拆分内容并告知用户
**在已有幻灯片中添加图片时：** 先将图片移到新幻灯片或减少其他内容。绝对不要在不检查现有内容是否已占满视口的情况下添加图片。

## Phase 1：内容发现（新建/转换）

### 提问能力适配（按平台选择）

需要让用户做结构化选择时，优先使用当前平台的原生提问能力，而不是把选项写成长文本让用户手动回复。

**公共原则（所有平台通用）**：
- 长内容先用普通消息展示，保持可读、可滚动、可复制；原生提问框只承载短问题和短选项。
- 原生能力不可用时，必须明确说明降级原因，再改用普通聊天短问。
- **禁止自行默认选择**：用途、篇幅、密度、浏览方式、风格、是否进入分享/导出等关键选择，必须由用户确认。除非用户已经明确给出偏好，否则不要因为工具不可用而替用户选择。
- 平台差异只影响“怎么问”，不影响“必须让用户选”的规则。

| 平台 | 优先能力 | 使用规则 |
|------|----------|----------|
| Claude Code | `AskUserQuestion` | 问题正文保持短句；不可用时普通短问并说明降级 |
| Antigravity | `ask_question` | 使用原生选项框；不可用时普通短问并说明降级；禁止把大纲、表格、长列表塞进问题或选项 |
| Codex | `request_user_input` | 使用 1-3 个短问题；选项必须短，推荐项放第一；如果当前模式下不可用，提示用户切到 Plan 模式完成选项选择，选完后提示用户切回执行模式继续生成 |
| Cursor | 可用工具不稳定 | 如果当前工具列表出现 `AskQuestion`、`ask_question` 或等价能力则使用；否则退回普通聊天短问，不伪造工具名 |
| 其他环境 | 普通聊天短问 | 一次只问必要问题，选项不超过 4 个 |

**统一原则**：长内容先展示，短问题再确认；选项只承载选择，不承载正文。

### 模式 A/C：新建/MD文档转换

**通过当前平台的原生提问能力一次性提出以下问题**，让用户一次性填写：  **问题 1 — 用途**（标题："用途"）： 这份汇报的用途是什么？选项：工作汇报 / 项目演示 / 教学分享 / 数据展示  **问题 2 — 篇幅**（标题："篇幅"）： 大约多少张幻灯片？选项：精简 5-10 / 中等 10-20 / 长篇 20+  **问题 3 — 内容**（标题："内容"）： 内容准备到什么程度？选项：内容已就绪（粘贴内容/MD文件） / 有要点大纲 / 只有主题方向  **问题 4 — 密度**（标题："密度"）： 这份汇报的内容密度应该是？选项：
- "低密度 · 演讲导向" — 一页一个核心观点，大字短语，视觉层次鲜明，适当增加页数
- "高密度 · 阅读导向" — 每页信息量更充实，结构化布局（表格/双栏/密集卡片），适合异步阅读和详细评审
**不主动询问浏览器内编辑** — 编辑功能默认包含。仅当用户明确要求"不要编辑功能"或"锁定导出"时才移除。  **记住用户的密度选择 — 它影响排版策略、字号大小、每页文字量和幻灯片数量。交互等级自动推断：低密度 → 标准交互（弹窗 + hover），高密度 → 标准 + 数据展示增强（tooltip 等）。如用户有明确的交互需求再调整。**  **模式 C（MD 文档转换）追加一问：**  **问题 5 — 浏览方式**（标题："浏览方式"，仅模式 C 时提出）： 偏好哪种页面浏览方式？选项：
- "翻页幻灯片（推荐）" — 适合投屏演讲、正式汇报
- "连续滚动" — 适合在线浏览、长文档、邮件分享  如果用户有内容，请他们分享（粘贴文字或提供 MD 文件路径）。

#### MD 文档内容识别（模式 C 专属）

当处理 Markdown 文档时，在进入 Phase 2 之前完成以下内容分析：
1. **标题层级扫描** — 将 `#`、`##`、`###` 映射为幻灯片标题/副标题
2. **表格识别** — 找出所有 Markdown 表格，标记列数和行数，评估是否需要拆页
3. **图片识别** — 找出所有 `![描述](路径)` 语法：
- 检查图片文件是否实际存在
- 根据文件名和上下文自动分类（架构图/流程图/截图/图表）
- 标记缺失的图片
4. **内容密度评估** — 评估每部分内容适合哪种排版策略（表格页/图文混排/稀疏内容/卡片网格）
5. **幻灯片大纲生成** — 基于标题层级和内容分析，自动生成初步的幻灯片大纲
6. **大纲确认** — 先用普通消息以 Markdown 列表展示完整大纲，保持可读、可滚动、可复制；随后再通过当前平台的原生提问能力请用户确认。问题正文必须短，不要把完整大纲、表格或长列表塞进问题或选项里。固定写法：
   - 标题："大纲"
   - 问题："这个大纲是否可以进入风格预览？"
   - 选项：确认使用 / 需要调整

### 图片评估（如果有图片）

如果用户提供了图片：
1. **扫描** — 列出所有图片文件（.png, .jpg, .svg, .webp 等）
2. **查看每张图片** — 使用 Read 工具查看（多模态读取）
3. **评估** — 对每张图片：描述内容、标记 可用/不可用（说明原因）、对应什么概念、主色调
4. **协同设计大纲** — 图片和文字共同决定幻灯片结构。不是"先规划幻灯片再加图片"，而是从一开始就围绕两者设计。
5. **确认** — 先用普通消息展示"幻灯片大纲 + 图片安排"，不要塞进原生提问框。随后通过当前平台的原生提问能力确认（标题："大纲"）：问题："这个大纲是否可以进入风格预览？" 选项：确认使用 / 需要调整

## Phase 2：风格选型

**这是"所见即所选"阶段。** 大多数人无法用文字准确描述自己的审美偏好。

### Step 2.1：情感倾向（引导式发现）

提问（标题："氛围"，多选，最多 2 项）： 希望听众在看这份汇报时的感受是？选项：
- 自信/有说服力 — 专业、可信赖
- 兴奋/有活力 — 创新、大胆
- 沉稳/专注 — 清晰、深思熟虑
- 受启发/被打动 — 有感染力、令人难忘

### Step 2.2：生成 3 套风格预览

根据情感倾向，生成 3 套不同的单页 HTML 预览，展示字体、配色、动画和整体氛围。**优先读取 [style-presets/index.json](references/style-presets/index.json) 了解 25 套预设的摘要和文件路由，再只读取本轮要使用的 3 个具体预设文件**。前 12 套按 `legacy-compatible` 规则继续使用；新增预设的来源与许可证见 [第三方预设声明.md](references/style-presets/第三方预设声明.md)。旧版 [风格预设.md](references/风格预设.md) 仅作为兼容入口和字体回退/CSS 注意事项参考，不应在 Phase 2 默认整份读取。
**预览组合规则（3 套预览 = 1 安全 + 1 大胆 + 1 创意）：**
- **1 套安全预设**：来自 12 套风格预设，与用户场景和情感最匹配的稳妥选择
- **1 套大胆混搭**：可以跨预设混搭配色和排版元素，或采用更鲜明的视觉对比（如从预设 A 取配色、预设 B 的字体、预设 C 的布局）
- **1 套创意 wildcard**：基于用户内容即兴设计。字体必须使用预设字体库中的字体，配色可以从现有预设中混搭，布局可自由组合  **预览真实性规则（不可妥协）：**  每个预览必须像可投屏的真实首屏，而非诊断卡片。
- 幻灯片上**禁止渲染**：`风格 A/B/C`、`选项`、预设名称、`preview`、文件路径、设计说明、`Option A/B/C` 等任何内部工作流文字
- 风格名称**仅在对话消息中**告知用户，不要渲染在幻灯片 HTML 里
- 幻灯片上只允许出现真实的汇报内容：标题、副标题、日期、汇报人、公司名等
- 如果幻灯片上出现了任何元数据标签，生成后立即修正
- 风格名称仅出现在给用户的对话消息中，不渲染在预览 HTML 里
| 情感倾向 | 推荐预设 |
|----------|----------|
| 自信/有说服力 | 大胆信号、电光工作室、暗夜植物园 |
| 兴奋/有活力 | 创意电压、霓虹赛博、双色粉彩 |

**预览文件与权限策略：**
- 优先将预览保存到 `.frontend-slides/slide-previews/`（style-a.html、style-b.html、style-c.html），每个预览文件应为自包含的单页 HTML，50-100 行，展示一个带动画的标题幻灯片。
- **本机浏览器优先规则（所有平台通用）**：用户验收预览时，必须优先打开到用户本机浏览器或系统默认浏览器；IDE 内置浏览器只作为自动化检查或本机浏览器不可用时的 fallback，不作为用户验收主入口。
- 打开优先级：平台 Chrome/系统浏览器控制能力 > 系统默认打开命令 > 完整本地 URL/路径说明 > IDE 内置浏览器 fallback。
- 系统默认打开命令：macOS 使用 `open <url-or-file>`；Windows 使用 `start <url-or-file>`；Linux 使用 `xdg-open <url-or-file>`。
- 预览和 demo 都应优先通过本地 URL 或本地文件路径打开；失败时必须说明失败原因，并给出完整路径/URL。
- **Antigravity 预览打开规则**：在 Antigravity 中，生成 3 个预览文件后，必须按上述本机浏览器优先规则尝试一次性自动打开 3 个预览，再进入风格选择。推荐只执行一次命令，降低权限打扰：
```bash
open .frontend-slides/slide-previews/style-a.html .frontend-slides/slide-previews/style-b.html .frontend-slides/slide-previews/style-c.html
```
- 如果自动打开失败，必须说明失败原因，并给出 3 个预览文件的完整路径，提示用户手动打开查看；不能在未打开或未提供可打开路径时直接进入风格选择问题。
- 其他平台如果会对 `mkdir`、`open`、`rm` 等命令逐条申请权限，优先减少 shell 命令：只生成必要文件；平台允许无打扰打开到本机浏览器时再自动打开。
- 如果无法创建多个预览文件，则生成一个包含三段预览区的自包含 HTML 文件作为替代，并清楚标注可选风格。

## Phase 3：生成汇报

### Phase 3 前置：内容完整性保障（不可跳过）

以原始 MD 文档为唯一内容来源，不省略、不摘要、不改写任何文字。图片 URL、表格、代码块、列表项均须完整保留。
**生成后自检（写入 HTML 之前执行）：**
1. 逐一核对 MD 原文中的所有 `![...](url)` 图片路径 — 每个 URL 必须完整出现在 HTML 中，丢失则立即补回
2. 逐一核对 MD 原文中的所有 `##` 标题文本 — 必须逐字出现在 HTML 的 kicker 或 h2 中
3. 检查 HTML 中的 `<table>` 数量是否与 MD 原文的表格数量一致
**生成前，务必按用户选择依次读取以下参考文件：**
| 内容密度 | 需要读取的参考文件 |
|---------|-------------------|
| 低密度 · 演讲导向 | [html模板.md](references/html模板.md) + [视口适配.css](references/视口适配.css) + [动画模式.md](references/动画模式.md) + [排版规范.md](references/排版规范.md) + [style-presets/index.json](references/style-presets/index.json) + 选定风格的具体预设文件 + [交互模式.md](references/交互模式.md)（仅模式1：弹窗） |
| 高密度 · 阅读导向 | 低密度所需参考文件 + [交互模式.md](references/交互模式.md)（模式1-3：弹窗、联动聚焦、tooltip） |
**关键要求：**
- 单个自包含 HTML 文件，所有 CSS/JS 内联
- 在 `<style>` 块中包含 [视口适配.css](references/视口适配.css) 的完整内容
- 使用 Fontshare 或 Google Fonts 的字体 — 禁止只用系统字体，必须包含中文字体回退
- **所有章节开头使用 section-head 标准模式**（kicker + h2 + note 三段纵向左对齐，note 放在大标题下方，不放右栏；参见排版规范.md）
- **Breadcrumb 章节导航由 JS 动态生成** — 在 `<style>` 中引用 html模板.md 的 `.breadcrumb` CSS 样式，在 `<script>` 中包含 `setupBreadcrumbs()` 和 `updateBreadcrumbs()` 方法。**禁止在 HTML 中手写 `<div class="breadcrumb">`**（会导致每张幻灯片重复渲染导航条）。JS 会自动从各 slide 的 `.section-kicker` 提取章节名并在 `<body>` 顶部创建唯一导航条。
- 为每张幻灯片选择 [排版规范.md](references/排版规范.md) 中定义的对应排版策略或预设组件
- **所有卡片/面板必须应用 hover 增强效果**（`@media (hover: hover)` 包裹，参见 html模板.md）
- **所有多卡片 grid 容器必须应用 `:has()` 联动聚焦**（参见交互模式.md）
- **交互声明必须真实落地**：当 HTML 中写到"弹窗"、"tooltip/提示"、"轮播/证据切换"等交互能力时，必须提供至少一个可操作实例和对应 JS/CSS；如果当前内容不需要该交互，就不要在页面中宣称它。
- **交互模式示例页要求**：如果生成专门介绍交互模式的页面，至少包含一个可点击弹窗、一个 tooltip 示例、一个轻量轮播/证据切换示例；否则删除这些能力声明，只保留实际实现的交互。
- **交互示例视觉约束**：交互能力可以通过小型 inline demo、弹窗内 demo、紧凑控件证明，不能把大面积黑色演示框塞进 slide 主体。轮播/证据切换示例默认高度不得超过 `min(22vh, 180px)`；若页面已有 5 张交互卡片，真实轮播示例优先放入弹窗，或压缩为一行控件，避免主体内容溢出。
- 为每个区块添加清晰的注释说明
- 每个区块用 `/* === 区块名称 === */` 注释块分隔
- **表格必须使用左对齐**，表头有底部分隔线
- **图片必须加点击放大功能**（lightbox）
- **内容较少的幻灯片必须加 `center-v`（硬性规则）** — 垂直居中，避免贴顶留白。判断标准：**每张 slide 生成前单独评估内容密度**，满足以下任一条件必须加 `center-v`：
- section-head + ≤ 4 张卡片（例如工作流优势、业务可行性）
- section-head + 一段话 + 短列表（≤ 5 项，例如项目目标、核心结论）
- section-head + 1 张图 + 简短 check-list（≤ 5 项，例如最新进展）
- section-head + 1-2 张 hero 图（无其他实质内容，例如核心流程图）

**反例（不加 center-v 的大密度页面）**：大表格（≥ 10 行）、双栏布局且两侧都密集（表格+代码块）、多个大 hero 图堆叠。

**生成后逐个 slide 复查**：如果某页内容看起来稀疏但没加 center-v，返回修改。尤其是 section-head + 1-4 张卡片的页面，不能只依赖全局 center-v 数量达标，必须逐页检查。
- **翻页幻灯片视觉平衡**：当用户选择"翻页幻灯片"时，每页主要内容必须在 100vh 内视觉居中或上下均衡分布；稀疏页使用 `center-v`，双栏页使用垂直居中或视觉平衡布局，避免内容集中在视口上半部分。
- **保留原始标题文本（硬性规则）** — 章节 kicker 和 h2 的文本必须逐字复制 MD 原始标题。"审核规则的多样性"必须写"审核规则的多样性"，不能缩写为"审核规则"或改写为"多维审核覆盖"。违反此规则视为生成失败。
- **图片生成与交付分离** — 生成阶段可以暂时保留原始本地路径或外部 URL，方便内容覆盖检查；进入 Phase 5 后必须运行图片内联脚本，最终 HTML 不得残留相对路径、绝对路径、`file://` 或 HTTP/HTTPS 图片引用
- 根据用户选择的页面模式：翻页幻灯片→scroll-snap；连续滚动→body.scroll-mode+sticky导航

## Phase 5：交付
1. **清理** — 如需清理 `.frontend-slides/slide-previews/`，优先在可无打扰执行命令的平台中进行；如果当前平台会逐条申请权限，可以跳过清理并在总结中说明预览目录位置，避免额外权限弹窗。
2. **图片内联（强制）** — 保存 HTML 后、质量验证前执行：
```bash
python scripts/inline_images.py <html文件路径> --base-dir <原始内容或HTML所在目录>
```
- 相对、绝对、`file://` 和 HTTP/HTTPS 图片必须转换为 `data:image/*;base64,...`。
- 下载失败、非法 MIME、私网地址或残留图片引用时阻塞交付，并通过 C05 让用户选择重试、替换、压缩或移除。
- 单图超过 10MB 或最终 HTML 超过 50MB 时必须通过 C05 确认；用户明确允许后才可加 `--allow-large` 重跑。
- 脚本失败时不得继续后续步骤，不得静默保留外链。
3. **运行质量验证（强制）** — 图片内联成功后立即执行：
```bash
python scripts/validate.py <html文件路径>
```
- **通过（≥80 分）** → 进入步骤 4；不要求自动修到 100 分。**硬停规则**：一旦达到 ≥80 且没有内容覆盖缺失，必须停止自行修复，先打开或定位 demo 让用户检查。
- **未通过（<80 分）** → 根据脚本输出中的失败项逐一修复 HTML，修复后**重新运行验证**，直到 ≥80 分。不允许跳过此步骤。
- **≥80 但 <100 分** → 不要自行继续修复到满分。先通过 C06 询问；只有用户明确选择“继续修复”时，才进入下一轮修复。
4. **运行内容覆盖率报告（Markdown 主路径强制）** — 如果输入来源是 `.md`，在 `validate.py` 之后继续执行：
```bash
python scripts/content_coverage.py <原始.md> <html文件路径>
```
- **通过** → 可以继续打开 demo。
- **有缺失** → 阻塞交付声明；必须回到原始 Markdown 和 HTML 修复缺失的标题、图片、表格、代码块或列表项，修复后重新运行覆盖率报告。
- **需复核** → 明确列出复核说明，不得直接宣称内容完整。
- **PPT/非 MD 路径** → 本轮自动内容覆盖率后置；使用 Phase 4 提取摘要和人工复核兜底，重点检查标题、图片、表格和备注是否进入 HTML。
5. **打开 demo（强制尝试）** — 质量验证 ≥80 且 Markdown 主路径内容覆盖率未发现缺失后，必须先帮助用户打开或定位 demo，再进入任何分享/导出流程。
   - 用户验收路径必须优先打开到用户本机浏览器或系统默认浏览器；IDE 内置浏览器只作为自动化检查 fallback。
   - 如果平台有 Chrome/系统浏览器控制能力，优先用 Chrome 或系统默认浏览器打开。
   - 如果平台允许无打扰打开，macOS 运行 `open [文件名].html`，Windows 运行 `start [文件名].html`，Linux 运行 `xdg-open [文件名].html`。
   - 如果平台需要权限确认，说明将打开本地 HTML，并请求/等待用户授权。
   - 如果不能打开，必须给出完整文件路径或本地 URL，并明确请用户手动打开检查。
6. **截图/人工验证** — 浏览器打开后，或在无法打开时通过文件检查，对至少 5 个非标题页进行检查：
- 图片是否存在且正确显示
- 表格是否完整、未溢出
- 排版是否符合对应的内容密度策略（无明显大面积留白）    若发现问题且分数 <80，直接修复；若分数 ≥80，先作为优化建议列出，等待用户确认是否修复。
7. **总结** — 告知用户：
- 推荐开头："恭喜，汇报 HTML 已经完成。"
- 文件位置、选用的风格名称、幻灯片数量、**验证得分**
- Markdown 主路径的内容覆盖率报告结论；如为 PPT/非 MD 路径，说明已使用提取摘要和人工复核
- 若未满 100 分，列出未通过项、影响判断、建议是否修复
- 导航方式：方向键、空格键、滚动/滑动手势、点击导航圆点
- 如何自定义：`:root` CSS 变量调整配色、字体链接更换字体、`.reveal` class 控制动画
- 浏览器内编辑默认开启：鼠标悬停左上角或按 E 键进入编辑模式，点击任意文字即可编辑，Ctrl+S 保存
- **如果内容包含图片**：图片支持点击放大查看
- **安全说明**：所有 JavaScript 仅用于页面导航、动画触发和编辑功能，不会采集数据、不会发送网络请求、不会访问剪贴板以外的权限。
8. **Demo 确认（Phase 6 前置门）** — 打开或定位 demo 后，必须通过当前平台的原生提问能力完成 C07。问题保持短句，不要把验证明细塞入问题框；原生能力不可用时按"提问能力适配"规则降级，不得自行默认进入 Phase 6：
   - 标题："Demo"
   - 问题："这个 demo 是否可以进入分享/导出？"
   - 选项：`可以进入分享/导出` / `需要继续修复`
   如果平台原生提问能力不可用，才退回普通聊天短问，并明确说明原因。用户选择“需要继续修复”时，参考 [排版规范.md](references/排版规范.md) 进行调整，修复后**重新运行 validate.py** 确认仍 ≥80；用户选择“可以进入分享/导出”时才进入 Phase 6。
9. **清理预览文件（进入 Phase 6 前）** — 用户确认 demo 可用后，再清理 `.frontend-slides/slide-previews/`。如果平台会逐条申请权限，可以询问是否清理；如果跳过清理，必须在总结中说明预览目录仍保留。
10. **进入 Phase 6（用户确认后衔接）** — 只有用户确认 demo 可进入分享/导出后，才进入 Phase 6。不要在用户尚未确认 HTML 有效时弹出 Phase 6 选项。

### 6B：导出 PDF

将每张幻灯片截图合成 PDF，适合邮件附件、嵌入文档或打印。
1. **运行导出脚本：**
```bash
bash scripts/export-pdf.sh <html文件路径> [输出.pdf]
```
如不指定输出路径，PDF 保存在 HTML 文件旁。
2. **说明**：首次运行可能需 30-60 秒（安装 Playwright 和 Chromium 浏览器）。后续导出会更快。

## 脚本失败处理（fallback 方案）

当脚本执行失败时，不要直接放弃——按以下 fallback 策略处理：
| 脚本 | 常见失败原因 | fallback 方案 |
|------|------------|--------------|
| `validate.py` | Python 版本过低 / 文件编码异常 | 改为**人工检查**：用 Grep 工具在生成的 HTML 中搜索 `100vh`、`center-v`、`lightbox`、`section-head`、`clamp(`，确认至少找到 5 项后再交付 |
| `extract-pptx.py` | `python-pptx` 未安装 | 提示用户安装：`pip install python-pptx`。若用户无法安装，**改为人工提取**：让用户打开 PPT，把每页的标题、正文、图片说明复制到一份临时 MD，然后用模式 C（MD 转换）代替 |
| `deploy.sh` | Node.js 未安装 / 未登录 Vercel | 脚本本身会给出详细引导。若仍失败，告知用户**手动部署**：把 HTML 文件直接通过微信/邮件发送给同事，或上传到公司内部文档系统 |
| `export-pdf.sh` | Playwright 安装失败 / Chromium 下载失败 | **改为浏览器打印 PDF**：告知用户在浏览器中按 `Ctrl+P`（或 `Cmd+P`），选择"另存为 PDF"，纸张设为"横向"，缩放设为 100% |  **通用原则**：
1. 脚本失败时，**必须告知用户失败原因和具体 fallback 步骤**，不要默默跳过
2. fallback 方案的产物质量**不会低于 80 分**（validate.py 的人工检查版本覆盖核心项目）
3. fallback 方案执行后，**仍需询问用户是否满意**，不满意则进入修改循环
